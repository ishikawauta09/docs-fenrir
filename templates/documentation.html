Introduction to Fenrir Framework

Fenrir is a next-generation, high-performance hybrid Python web framework built on top of modern ASGI (Asynchronous Server Gateway Interface) specifications. The fundamental problem Fenrir solves is ecosystem fragmentation. Historically, developers were forced to choose a single design pattern at the inception of a project: choosing Flask for quick, imperative monolithic web apps, FastAPI for strict type-hinted data validation, or Falcon for low-overhead, object-oriented REST architectures.

Fenrir breaks down these architectural silos by gracefully unifying the absolute best design patterns from Python’s most successful web frameworks into a singular, cohesive, and high-performance workspace:

  • Flask-Style: Offers seamless, imperative data manipulation via thread/task-safe Context Locals, highly intuitive route decorators, and native integration with the Jinja2 dynamic templating engine for traditional Server-Side Rendered (SSR) layouts.
  • FastAPI-Style: Provides automatic and ultra-fast request body parsing/validation using Pydantic v2, strict static type hinting compatibility, and an elegant, composable Dependency Injection (DI) engine.
  • Falcon-Style: Enables programmatic, enterprise-grade REST compliance using Class-Based Resources, accompanied by direct low-level manipulation of explicit Request and Response objects and pre/post-execution hooks.
  • Sanic-Style: Empowers responsive server lifecycle management via asynchronous event listeners (e.g., before/after startup hooks) and effortless background task scheduling directly onto the main event loop.
  • Bottle-Style: Preserves backward compatibility by allowing legacy WSGI synchronous web applications to be mounted and executed smoothly under isolated sub-URL prefixes.

All of these diverse programming paradigms do not run in isolation; rather, they are orchestrated concurrently atop the ultra-optimized Asteri v2.2.2 ASGI engine, guaranteeing maximum network throughput and negligible internal abstraction overhead.

Why Embrace a Hybrid Architecture?

In large-scale microservice architectures or sophisticated enterprise monoliths, different sub-domains demand different tooling. A user-registration API gate requires strict schema validation (FastAPI-style); a public-facing billing landing page demands dynamic HTML compilation (Flask-style); and an IoT data-ingestion pipeline requires pure object-oriented class resource logic (Falcon-style). Fenrir eliminates the need to split your infrastructure into separate microservices just to accommodate writing styles, dramatically cutting down your server memory footprint and code management complexity.

Key System Features

  • ⚡ Ultra-Fast ASGI Core: Specifically engineered to eliminate stack-frame overhead. The internal router uses highly optimized tree-lookup structures to process high-concurrency connections with single-digit millisecond latency.
  • 🧩 Multi-Paradigm Hybridization: Mix and match declarative Pydantic schemas and imperative context globals within the same file, or even within adjacent routes, without any memory safety issues.
  • 📖 Native OpenAPI Document Automation: Upon server boot, Fenrir reflects upon your route signatures and type declarations. It automatically compiles an active OpenAPI schema, exposing live interactive Swagger UI (at /docs) and ReDoc without any manual YAML authoring.
  • 🔌 Built-in Real-Time Protocols: Full kernel-level orchestration for bidirectional WebSockets and unidirectional Server-Sent Events (SSE) out of the box, requiring zero external adapters.
  • 🛠️ Premium CLI Automation Suite: Packed with professional developer tooling to automate directory scaffolding, run isolated zero-network micro-benchmarks, and visualize complex routing structures.

Installation & Setup

Fenrir adheres to a strict minimalist dependency philosophy to guarantee that container initialization times (e.g., in Docker layers) remain instantaneous and baseline RAM allocation stays as lean as possible. The core runtime requires **Python 3.8 up to the latest Python 3.13**.

1. Production Installation (via PyPI)

This standard installation downloads the latest compiled stable binary distribution of Fenrir along with all necessary production-ready infrastructure modules.

pip install fenrir-framework

2. Development Installation (From Source Code)

If you intend to patch internal engine mechanics, contribute to the open-source repository, or test bleeding-edge nightly features, install the repository in editable mode.

git clone https://github.com/IshikawaUta/fenrir.git
cd fenrir
pip install -e .

Core Dependency Analysis

Here is why Fenrir locks specific versions of the following architectural components into its system core:

  • pydantic >= 2.0.0: Leverages Pydantic v2's Rust-compiled core engine to transform incoming stringified JSON streams into native validated objects at hyper-speed.
  • jinja2 >= 3.0.0: Provides a robust sandboxed HTML rendering ecosystem equipped with automatic escaping to eliminate Cross-Site Scripting (XSS) risks.
  • asteri >= 2.2.2: Acts as the low-level ASGI engine that intercepts networking sockets, aggregates incoming TCP byte arrays, and manages connection pools asynchronously.
  • itsdangerous >= 2.0.0: Handles cryptographic token signing and secure cookie encryption to completely block client-side state spoofing.
  • python-multipart >= 0.0.18: A streaming multipart parser that splits incoming binary data chunks during large file uploads, preventing out-of-memory crashes.

Quick Start

The following minimum reproducible sample showcases how effortlessly three completely different development philosophies (Flask, FastAPI, and Falcon) mesh together inside a single app.py execution file: 

from fenrir import Fenrir

# Initialize the main app instance and establish fundamental metadata for automatic docs
app = Fenrir(title="My First Application", version="1.0.0")

# [Flask/FastAPI Style]: Simple, declarative asynchronous handler using functional routing decorators
@app.get("/")
async def home():
    return {"message": "Welcome to Fenrir!"}

# [FastAPI Style]: Path parameters are strictly cast and validated inline against native type hints
@app.get("/users/<user_id:int>")
async def get_user(user_id: int):
    return {"user_id": user_id, "name": "John Doe"}

# [Falcon Style]: Clean, class-based resource routing enforcing state isolation
class ItemResource:
    async def on_get(self, req, resp, item_id: int):
        # Manipulate the explicit response instance independently without violating state rules
        resp.media = {"item_id": item_id, "status": "active"}

# Register the standalone resource class to the central routing tree
app.add_route("/items/<item_id:int>", ItemResource())

if __name__ == "__main__":
    app.run()

Invoke the native Fenrir CLI binary helper to spawn an active listener thread. Passing the --dev flag arms an active file-system monitor that triggers hot-reloads instantly whenever code modifications are saved. 

fenrir run app:app --port 8088 --dev

Basic Concepts

Central Application Instantiation

The root Fenrir instance acts as the command center of your system. Critical parameters mapping security rules, title metadata, and precise endpoints for autogenerated schema renderers are passed here during boot initialization. 

from fenrir import Fenrir

# Comprehensive configuration dictating the live mapping locations of your autogenerated docs
app = Fenrir(
    title="My Application",
    version="1.0.0",
    description="An amazing application",
    docs_url="/docs",           # Interactive Swagger UI route designed for frontend engineers
    redoc_url="/redoc",         # Static schema visualization designed for long-form technical review
    openapi_url="/openapi.json" # Raw JSON output holding your standard compliant spec metrics
)

Routing

Fenrir's routing tree parses structural arguments straight from URL pathways. When an incoming network packet strikes the server, the routing parser dissects strings, automatically casts variables to your declared target types, and injects them directly into your handler inputs.

Path Converters & Query String Validation Rules

  • <param>: Standard string capture mechanism. It extracts path slices up until the next forward slash.
  • <param:int>: Evaluates if a segment is an integer. If non-numeric strings are received, the request skips the handler and drops out into a `404 Not Found` state.
  • <param:path>: Grabs everything remaining in the tail end of the URL string including embedded slashes, which is essential for file streaming handlers.
# Extracts standard text patterns out of a route path
@app.get("/users/<username>")
async def get_user(username: str): pass

# Forces numerical parameter matching and type casting
@app.get("/posts/<post_id:int>")
async def get_post(post_id: int): pass

# Captures exhaustive nested directory pathways raw
@app.get("/files/<file_path:path>")
async def get_file(file_path: str): pass

# Incorporating explicit 'Query' bounds to protect your server against excessive offset bounds (?skip=0&limit=10)
from fenrir import Query

@app.get("/items")
async def list_items(
    skip: int = Query(0, ge=0),       # Defaults to 0, enforces a value greater than or equal to (ge) 0
    limit: int = Query(10, le=100)    # Defaults to 10, Caps extreme reads at less than or equal to (le) 100
):
    return {"skip": skip, "limit": limit}

Request & Response

Fenrir gives you two paths for data egress and ingress management: an **imperative global proxy** (importing the thread-isolated `request` context directly) and **explicit object construction** via custom Response classes.

Returning raw Python dictionary types automatically passes through Fenrir’s serialization pipeline, adding an `application/json` Content-Type header. However, declaring an explicit response wrapper lets you configure headers and statuses down to the byte level for optimal network performance. 

from fenrir import request, JSONResponse, HTMLResponse, RedirectResponse

@app.get("/info")
async def get_info():
    # Inspects HTTP headers imperatively from the isolated local request proxy
    user_agent = request.headers.get("User-Agent")
    return {"user_agent": user_agent}

@app.get("/explicit-json")
async def explicit():
    # Explicit response initialization with customized exit status codes
    return JSONResponse({"key": "value"}, status_code=200)

Dependency Injection

Fenrir’s *Dependency Injection* (DI) engine resolves prerequisite graphs before a request ever hits your target route handler. This approach separates boilerplate tasks (like extracting API security keys, parsing JWT authorization signatures, or establishing database connection transactions) from your core business route handling logic.

If a sub-dependency task fails validation and raises an `HTTPException`, Fenrir completely halts the execution pipeline at that exact phase. It sends the error response back to the client immediately, protecting your internal business handlers from running on corrupted or unauthorized states. 

from fenrir import Depends, Header, HTTPException

# Prerequisite authorization token validation dependency
async def verify_token(x_api_key: str = Header(default=None)):
    if x_api_key != "secret-token":
        raise HTTPException(status_code=401, detail="Invalid credentials")
    return x_api_key

# Injecting prerequisite steps into path variables using the 'Depends' token wrapper
@app.get("/protected")
async def protected_route(token: str = Depends(verify_token)):
    # This block is guaranteed to execute only if 'verify_token' evaluates successfully
    return {"auth": token}

FastAPI Paradigm (Data Integrity & Validation)

The FastAPI paradigm in Fenrir relies heavily on native Python **Type Hinting** to manage incoming payloads automatically. When an incoming HTTP request containing a POST payload strikes an active endpoint, Fenrir acts as an validation interceptor before activating your handler.

The raw JSON buffer stream is ingested asynchronously, parsed, and evaluated against the data structures declared in your **Pydantic v2** models. If the client delivers malformed fields (such as text values where integers are required, or a password failing minimal character bounds), Fenrir halts the pipeline. It bypasses your handler completely and returns a clear `422 Unprocessable Entity` response, highlighting exactly where the data validation failed before it can corrupt down-stream processes. 

from pydantic import BaseModel, Field
from fenrir import Fenrir, Depends, Header

app = Fenrir()

# Establish strict declarative blueprints defining structural input integrity expectations
class UserRegister(BaseModel):
    username: str = Field(..., min_length=3) # Mandatory field, rejecting inputs short of 3 characters
    email: str                               # Automatic pattern validation for email addresses
    age: int                                 # Enforces valid integer casting bounds

async def verify_token(x_api_key: str = Header(default=None)):
    if x_api_key != "secret-token":
        raise ValueError("Invalid credentials")
    return x_api_key

# Hybrid Execution: Merging direct Pydantic structural data checks alongside active dependency routines
@app.post("/api/register")
async def register(body: UserRegister, token: str = Depends(verify_token)):
    # At this threshold, the 'body' instance is guaranteed to be clean, trusted, and verified
    return {"status": "success", "user": body.model_dump(), "auth": token}

Flask Paradigm (Asynchronous Context Locals)

One of the most persistent bottlenecks when developing within pure asynchoronous `async/await` lifecycles is thread-state sharing. Passing instance records (like active Request payloads, User profile blocks, or Transaction states) across nested service layers often forces developers to include these objects as messy arguments in every single method call.

Flask solves this in synchronous execution models using *Thread Locals*. Because asynchronous engines run concurrent coroutines over a singular unified thread layer, Fenrir ports this pattern over by utilizing advanced **`contextvars`** architectures.

Context Isolation Under Concurrency

Every time a distinct coroutine is scheduled by the Asteri ASGI event loop, Fenrir provisions an isolated context-variable storage tokens frame. This guarantees that imported global states like request and local data registries like g remain insulated against cross-user race conditions, even under extreme volumetric loads.

Thanks to this architecture, you gain the complete structural agility of Flask. You can pull query variables out of global space imperatively, deposit custom global states into the `g` instance on a per-request basis, and compile dynamic web content on the fly via Jinja2 engine integrations. 

from fenrir import Fenrir, request, g, render_template

app = Fenrir()

@app.get("/welcome")
async def welcome():
    # Pull parameters imperatively out of the active contextual request proxy frame
    name = request.args.get("name", "Guest")
    
    # 'g' is an isolated sandbox that persists exclusively for the lifespan of this single request
    g.visit_count = g.get("visit_count", 0) + 1
    
    # Compile dynamic web structures server-side while infusing local state tracking values
    return render_template("welcome.html", name=name, count=g.visit_count)

Falcon Paradigm (Class-Based Resources)

As enterprise backends scale, mapping complex endpoints with functional route decorators can scatter code logic everywhere. This fragmentation happens because your GET, POST, PUT, and DELETE operations are handled by separate, independent functions. Fenrir addresses this by introducing pure **Class-Based Resource** per-route bindings, modeled after Falcon's design patterns.

Under this pattern, your routes are linked to a single, unified class instance rather than separate functions. The class methods are strictly named after HTTP verbs with an `on_` prefix (such as on_get, on_post, and on_delete). This architectural separation keeps incoming request properties (`req`) isolated from outgoing response structures (`resp`), giving developers precise, low-level control to manipulate header states or set status codes explicitly. 

from fenrir import Fenrir

app = Fenrir()

class ItemResource:
    # 'req' controls input ingestion, while 'resp' acts as an isolated outbox canvas
    async def on_get(self, req, resp, item_id: int):
        resp.status = 200
        resp.media = {"item_id": item_id, "msg": "Fetched via Falcon Class-Based view!"}

    async def on_post(self, req, resp, item_id: int):
        # Access incoming JSON values straight through the structured request object
        data = req.json
        resp.status = 201
        resp.media = {"item_id": item_id, "received": data}

# Attach all unified verb routines to a single endpoint node on the master application instance
app.add_route("/items/<item_id:int>", ItemResource())

File Upload

Handling binary asset ingestion in Fenrir utilizes an asynchronous streaming generator pattern. This design prevents large multi-gigabyte files from causing sudden memory spikes that could compromise other operations running on your server.

By processing data chunks using the UploadFile wrapper, Fenrir avoids loading entire files into RAM at once. Developers can stream incoming data slices via `.read()`, monitor file sizes on the fly, and securely pipe data directly to disk arrays or cloud storage nodes. 

from fenrir import File, UploadFile

@app.post("/upload")
async def upload_file(file: UploadFile = File(...)):
    # Stream binary content out via non-blocking asynchronous chunks
    contents = await file.read()
    return {"filename": file.filename, "size": len(contents)}

WebSocket Communication

Standard HTTP operations follow a strict, unidirectional model—stateless request-response patterns. For highly interactive, full-duplex systems like live chats or real-time geolocation tracking, Fenrir offers a native @app.websocket() decorator.

When a WebSocket handler is activated, it performs a protocol upgrade handshake to step up from regular HTTP. By using a standard `try-except` block to capture `WebSocketDisconnect` exceptions, you can easily track live connection pools, handle streaming strings or binary frames, and broadcast states across connected nodes without blocking the main loop. 

from fenrir import WebSocket, WebSocketDisconnect

@app.websocket("/ws/echo")
async def websocket_echo(websocket: WebSocket):
    # Intercept and process the client's protocol upgrade handshake request
    await websocket.accept()
    try:
        while True:
            # Wait for incoming text frames without blocking the main server loop
            data = await websocket.receive_text()
            # Push message echo instantly down the established socket pipe
            await websocket.send_text(f"Echo: {data}")
    except WebSocketDisconnect:
        # Perform necessary cleanups or remove active client reference markers upon logout
        print("Disconnected")

Server-Sent Events (SSE)

While WebSockets are excellent for heavy, bidirectional communication, **Server-Sent Events (SSE)** provide a lightweight, resource-efficient alternative for unidirectional streams. This pattern is ideal when the server needs to push continuous updates down to the client without waiting for responses (e.g., streaming live system metrics, sports telemetry, or real-time token generation from large language models).

Fenrir implements this pattern via the EventSourceResponse class, which wraps an asynchronous generator function. The server holds the initial HTTP connection open and pushes a continuous stream of text chunks down to the client using standard formatting prefixes (`data: ...\n\n`) without the overhead of connection renegotiations. 

from fenrir import EventSourceResponse
import asyncio

@app.get("/events")
async def get_events():
    # Asynchronous generator tasked with feeding data continuously into the response pipe
    async def event_generator():
        for i in range(10):
            yield f"data: Message {i}\n\n" # Standard layout required by W3C EventSource specifications
            await asyncio.sleep(1)          # Relinquish loop controls for 1 second before streaming the next chunk
    return EventSourceResponse(event_generator())

Sanic Paradigm (Lifecycle Listeners & Background Tasks)

Enterprise web applications require dependable hook points to manage their underlying infrastructure lifecycles, along with the ability to offload long-running tasks away from the client response path. Fenrir supports this by incorporating **Event Listeners** and **Background Task Scheduling** inspired by Sanic.

Any function registered under the `before_server_start` listener hook is guaranteed to complete before the server begins accepting public network traffic, making it the perfect location to initialize database connection pools or load cache maps into memory.

For slow, resource-heavy operations (like firing out transactional emails, running image compression tasks, or processing telemetry logs), you can dispatch tasks directly into the background using add_task(). The client receives a success response in milliseconds, while the event loop handles the heavier work asynchronously in the background. 

import asyncio
from fenrir import Fenrir

app = Fenrir()

# Pre-startup listener: Establish target database connection instances before allowing ingress traffic
@app.listener("before_server_start")
async def setup_db_pool(app_inst):
    print("Database connection initiated!")

# Independent background loop tasked with collecting processing metrics
async def background_poller():
    while True:
        print("Polling metrics...")
        await asyncio.sleep(60) # Non-blocking execution sleep; safely repeats every 60 seconds

@app.listener("after_server_start")
async def run_poller(app_inst):
    # Inject our looping coroutine directly into the system's core async runtime task scheduler
    app_inst.add_task(background_poller())

Bottle Paradigm (WSGI Application support)

Migrating large, legacy codebases from synchronous WSGI models (such as older Flask frameworks, Django, or legacy Bottle services) to modern, asynchronous ASGI runtimes can be difficult, as it often requires rewriting extensive portions of the application. To ease this transition, Fenrir includes an advanced **WSGI Mounting Adapter**.

This adapter wraps legacy WSGI instances and runs them within a managed thread pool executor. From the perspective of Fenrir's ASGI core engine, these wrapped endpoints behave like standard, non-blocking asynchronous paths, allowing legacy services to run safely inside specific, isolated sub-URL prefixes. 

from fenrir import Fenrir, redirect
from legacy_app import flask_wsgi_app # Imports a classic synchronous Flask WSGI app instance

app = Fenrir()

# Mount the legacy app instance directly to an isolated sub-url prefix
app.mount_wsgi(flask_wsgi_app, prefix="/legacy")

@app.get("/old-docs")
async def old_docs():
    # Seamlessly route modern async traffic toward new infrastructure targets
    return redirect("/docs")

Error Handling & Exceptions Compatibility

Fenrir's centralized exception handling layer intercepts application failures and status errors using the @app.exception() decorator. This catches unhandled bugs before they bubble up, preventing raw stack traces from exposing sensitive information to end-users.

1. Intercepting Validation Failures (FastAPI Style)

Intercepts validation errors thrown by Pydantic v2 and restructures the output payload to match your team's standard JSON format.

from fenrir import JSONResponse

@app.exception(422)
async def validation_error_handler(req, exc):
    return JSONResponse(
        status_code=422,
        content={"error": "Validation failed", "errors": exc.detail if hasattr(exc, 'detail') else []}
    )

2. Custom Error Pages (Flask Style)

Intercepts standard 404 status codes globally to serve custom, user-friendly HTML error pages, ensuring a cohesive user experience.

from fenrir import render_template

@app.exception(404)
async def page_not_found(req, exc):
    # Render the custom static template while appending the native 404 status code in headers
    return render_template("404.html", detail=exc.detail), 404

Middleware (Flask & Sanic Style)

*Middleware* provides global interceptor hooks executed across the request-response lifecycle. Fenrir divides these hooks into two distinct execution phases:

  • @app.before_request: Triggers immediately after an incoming HTTP request is parsed, but before it reaches its target route handler. This phase is ideal for global task execution, such as traffic logging or IP address blocklists.
  • @app.after_request: Executes right after a route handler builds its response object, allowing you to inject custom security headers or modify payloads before the data travels over the network.
@app.before_request
async def log_request():
    # Log incoming request paths globally for real-time traffic monitoring
    print(f"Request Path: {request.path}")

@app.after_request
async def add_security_headers(response):
    # Inject clickjacking protection into all outgoing response headers automatically
    response.headers["X-Frame-Options"] = "DENY"
    return response

Authentication & Security

Securing your endpoints in Fenrir is simplified with built-in extractors for standard industry authentication schemes. Using the HTTPBearer wrapper class, Fenrir automatically checks for the presence of valid `Authorization: Bearer ` payloads within incoming request headers. If this value is missing or malformed, the request is blocked automatically with a `401 Unauthorized` status response, protecting your backend services from unauthenticated traffic. 

from fenrir import HTTPBearer, Depends

bearer = HTTPBearer()

@app.get("/secure")
async def secure_endpoint(credentials = Depends(bearer)):
    # Read the extracted bearer token value directly within your endpoint handler
    token = credentials.credentials
    return {"token": token, "status": "authenticated"}

Blueprints (Modular Routing)

**Blueprints** are an essential tool in Fenrir for partitioning large applications into clean, isolated modules. Instead of registering hundreds of routes directly onto your primary application instance, you can group related paths into individual sub-containers based on their business domain (e.g., authentication, checkout, or analytics).

Each Blueprint can manage its own configuration settings, such as appending automatic URL prefixes like `/api`. This keeps your routing architecture clean, scannable, and highly maintainable as the codebase scales. 

from fenrir import Blueprint, Fenrir

# Isolate all API transaction pathways into a modular sub-blueprint container
api_bp = Blueprint("api", url_prefix="/api")

@api_bp.get("/items")
async def list_items():
    # This route automatically resolves to the full path string: /api/items
    return [{"id": 1, "name": "Blueprint Item"}]

app = Fenrir()
# Register the isolated blueprint container onto the root app core coordinator
app.register_blueprint(api_bp)

Testing

Fenrir features a high-speed **TestClient** utility designed for integration testing. The test client utilizes a direct bypass mechanism powered by the HTTPX library to send requests directly into Fenrir's ASGI engine pipeline, eliminating the need to bind to real local network ports.

This design speeds up unit test suites within your continuous integration (CI/CD) pipelines, enabling quick validation of backend features before changes are deployed to live staging environments. 

from fenrir.testing import TestClient

def test_app():
    # Instantiate the virtual client wrapper enclosing the main application instance
    client = TestClient(app)
    # Fire an in-memory virtual request directly down into the ASGI pipeline loop
    response = client.get("/")
    
    # Assert structural responses strictly to prevent code regression anomalies
    assert response.status_code == 200
    assert response.json() == {"message": "Welcome to Fenrir!"}
